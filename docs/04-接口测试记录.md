## 阶段 23-修复：优惠券后台与前台展示回归问题

复核日期：2026-07-01

### 问题原因定位

- 后台新增优惠券：前端表单提交的创建参数需要与后端 `CouponCreateRequest` 保持一致，尤其是 `startTime/endTime` 的 `yyyy-MM-dd HH:mm:ss` 格式、金额字段、库存字段和状态字段；本次统一创建 payload，并增强时间校验。
- 后台删除已下线优惠券：后台列表继续复用公开 `GET /api/coupon/page` 时，公开列表一旦过滤下线券，后台就无法看到并操作下线券；本次新增后台分页 `GET /api/coupon/admin/page`，后台页面改为调用该接口。
- 前台仍显示已下线券：公开优惠券分页原先未按可领取状态过滤；本次只返回已上线、已开始、未过期、剩余库存大于 0、未逻辑删除的券。
- 部分商品无法加入购物车：前端存在 `id/productId` 字段来源不一致的商品卡片数据，可能传出空 `productId`；后端原先对商品下架或库存不足缺少写入前明确校验。本次前端统一解析商品 ID，后端加入购物车时明确校验商品状态和库存。

### 回归测试接口

```http
POST   /api/coupon
GET    /api/coupon/page
GET    /api/coupon/admin/page
PUT    /api/coupon/{id}/up
PUT    /api/coupon/{id}/down
DELETE /api/coupon/{id}
POST   /api/cart/add
```

### 页面检查点

- `/admin/coupons`：新增优惠券提交真实接口；后台列表能看到下线券；未被领取的下线券可删除。
- `/coupons`：只展示可领取优惠券，不展示下线、删除、未开始、已过期、库存为 0 的券。
- `/my-coupons`：已下线或已删除的用户券标记为“不可用 / 已失效”。
- `/products`、`/products/{id}`：加入购物车使用真实商品 ID，缺少 ID 时前端直接提示。
- `/cart`、`/checkout`：购物车和结算链路继续以后端接口为准。

### 删除规则复核

- 未被领取的优惠券，包括已下线未领取券，可以逻辑删除。
- 已被领取的优惠券不做物理删除，后端返回业务错误，避免破坏用户已领券和历史订单展示。
- 逻辑删除后的优惠券不会出现在前台公开优惠券中心。

## 阶段 21-5：真实接口响应结构复查

复查日期：2026-06-27  
入口地址：`http://localhost:9000`

### 复查接口与字段结构

`GET /api/home/index` 返回 `code=200`。真实字段结构如下：

- `categories`：数组，分类字段为 `id`、`name`、`parentId`、`sort`、`status`、`createTime`、`updateTime`。
- `hotProducts`：数组，商品字段为 `id`、`categoryId`、`categoryName`、`name`、`subtitle`、`mainImage`、`price`、`originalPrice`、`sales`、`status`、`stock`、`createTime`、`updateTime`。
- `recommendProducts`：数组，推荐商品字段为 `recommendId`、`recommendTitle`、`sort`、`productId`、`productName`、`subtitle`、`mainImage`、`price`、`originalPrice`、`sales`、`status`、`stock`、`createTime`、`updateTime`。
- 结论：`recommendProducts` 不返回 `id/name`，而是返回 `productId/productName`，前端必须通过统一 `normalizeProduct` 兼容该结构，否则会出现商品 ID 为 0 或“未命名商品”。

`GET /api/product/page?pageNum=1&pageSize=10` 返回 `code=200`，分页结构为 `total/pages/pageNum/pageSize/records`。`records` 商品字段为 `id`、`categoryId`、`categoryName`、`name`、`subtitle`、`mainImage`、`detail`、`price`、`originalPrice`、`sales`、`status`、`stock`、`lockedStock`、`createTime`、`updateTime`。

`GET /api/product/{id}` 返回 `code=200`。商品详情字段与商品分页基本一致，包含 `mainImage`。本次复查中，商品详情的 `mainImage` 已有 MinIO URL，而首页热门缓存中同一商品仍可能是空图，说明首页缓存和热门商品缓存需要清理后重新从商品服务读取最新字段。

`GET /api/coupon/page?pageNum=1&pageSize=10` 返回 `code=200`。优惠券分页字段为 `id`、`name`、`amount`、`minAmount`、`totalStock`、`availableStock`、`startTime`、`endTime`、`status`、`createTime`、`updateTime`。

`GET /api/coupon/my` 携带普通用户 token 返回 `code=200`。当前测试用户未领取优惠券时 `data=[]`；领取后应返回 `userCouponId`、`couponId`、`couponName`、`amount`、`minAmount`、`status`、`receiveTime`、`useTime`、`orderId`、`startTime`、`endTime`。

`GET /api/order/admin/page?pageNum=1&pageSize=10`：

- 当前 9050 运行实例经 Gateway 返回 `code=500`、`message=系统异常`。
- 使用当前源码已构建 jar 临时启动到 `19050`，直接请求 `/admin/page?pageNum=1&pageSize=10` 并传 `X-User-Id=39`、`X-User-Role=ADMIN` 返回 `code=200`。
- 管理员订单分页字段为 `id`、`orderNo`、`userId`、`totalAmount`、`discountAmount`、`payAmount`、`status`、`receiverName`、`receiverPhone`、`receiverAddress`、`payTime`、`deliveryCompany`、`trackingNo`、`deliveryTime`、`receiveTime`、`createTime`。
- 使用当前构建 jar 临时启动到 `19050` 后，调用 `POST /{orderId}/ship` 并传 `deliveryCompany`、`trackingNo` 返回 `code=200`，订单状态从 `1` 变为 `2`。
- 结论：后端源码和已构建 jar 中管理员订单分页接口可用；当前 9050 需要用最新构建重启 order-service 后再通过 Gateway 验证。

### 本阶段修复依据

- 首页热门商品按 `id/name/mainImage` 适配。
- 首页推荐商品按 `productId/productName/mainImage` 适配。
- 商品详情、商品分页、购物车、订单确认统一走商品字段归一化或等价字段适配。
- 首页真实图片替换后需要清理 `youxuan:home:index`、`youxuan:home:hot-products`、`youxuan:home:recommend-products` 和相关 `youxuan:product:detail:*`。
